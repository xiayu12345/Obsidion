# 按键发声蜂鸣与等待旋律开发记录

**日期**：2026-05-18  
**工程**：`bk7252n_sdk-SDK_3.0.76.1_witch_lcd`  
**编译命令**：`scons --beken=bk7252n -j4`

---

## 1. 背景

此前按键操作（单击/长按/松开）无任何声音反馈，用户体验生硬。为提升产品交互质感，需要：

- **按键按下时发出短促提示音**，让用户明确感知按键已被触发；
- **语音录音期间自动静音按键提示音**，避免"滴滴"声混入语音上传数据；
- **云端生图等待期间播放循环旋律**，替代死板的静默等待，增强陪伴感。

目标用户为 **6-10 岁儿童**，提示音需有"玩具感"而非生硬的电子蜂鸣。

## 3. 技术方案

### 3.1 统一常驻音频线程架构

采用 **单一常驻线程 + 命令/信号量** 模式，替代早期双线程方案：

- **线程名**：`"audio_th"`，栈 4096B，优先级 27
- **命令枚举**：`BEEP_CMD_NONE` / `BEEP_CMD_PLAY` / `BEEP_CMD_MELODY_START` / `BEEP_CMD_MELODY_STOP`
- **调度方式**：各 API 写入 `s_cmd` 后 `rt_sem_release` 唤醒线程；线程处理完后将 `s_cmd` 复位为 `BEEP_CMD_NONE`
- **初始化**：`beep_init()` 由 `main.c` 在系统启动阶段调用，创建信号量和线程，并预分配蜂鸣 PCM 缓冲区

**优势**：
- 消除双线程的栈开销（原 2×2048 = 4096B → 现单线程 4096B，实际节省了线程控制块等元数据）
- 命令互斥天然保证 DAC 不并发争抢
- 旋律停止时通过 `rt_sem_release` 立即唤醒（不再依赖轮询标志位延时）

### 3.2 "叮咚"双音蜂鸣

采用 **PCM 直灌 DAC** 方案，绕过 MP3 解码器，彻底规避栈溢出风险（此前 MP3 提示音曾因 `minimp3` 栈上大结构体导致 `auprompt` 线程栈溢出死机）。

**音色设计**：
- **高音段（"叮"）**：667Hz 正弦波 @ 8000Hz 采样率，12 采样点/周期，循环 55 周期 ≈ 83ms
- **低音段（"咚"）**：500Hz 正弦波 @ 8000Hz 采样率，16 采样点/周期，循环 45 周期 ≈ 90ms
- **总时长** ≈ 173ms，类似玩具门铃的"叮咚"效果，适合儿童产品

**PCM 数据**（编译期常量，存 Flash）：

```c
/* 667Hz 正弦波，12 采样点/周期 */
static const int16_t BEEP_PCM_HIGH[] = {
    0x0000, 0x4000, 0x6EDA, 0x7FFF,
    0x6EDA, 0x4000, 0x0000, 0xC000,
    0x9126, 0x8001, 0x9126, 0xC000,
};
/* 500Hz 正弦波，16 采样点/周期 */
static const int16_t BEEP_PCM_LOW[] = {
    0x0000, 0x30FB, 0x5A82, 0x7641,
    0x7FFF, 0x7641, 0x5A82, 0x30FB,
    0x0000, 0xCF05, 0xA57E, 0x89BF,
    0x8001, 0x89BF, 0xA57E, 0xCF05,
};
```

**播放流程**（在 `audio_th` 线程中执行）：
1. 收到 `BEEP_CMD_PLAY` 命令后，打开 `"sound"` DAC 设备，设置 8000Hz 采样率
2. 使用线程启动时预分配的 `beep_buf`（已填充高音段+低音段 PCM）
3. 保存并清除 `tx_complete` 回调，`rt_device_write` 一次性写入全部 PCM 数据
4. 延时 200ms 等待 DAC FIFO 排空
5. 恢复 `tx_complete` 回调，关闭 DAC

**资源占用**：
- Flash：约 56B（两段正弦波常量）
- RAM：`beep_buf` 约 2.4KB（线程生命周期内常驻），线程栈 4KB（与旋律共用）

### 3.3 等待旋律（C大调琶音）

**用途**：云端 AI 生图期间替代静默等待，循环播放至生图完成。

**旋律数据**：
- 预生成 C大调琶音 PCM，8000Hz mono int16_t
- 34400 采样 ≈ 4.3 秒/循环
- 存为 Flash 常量 `MELODY_PCM[]`（`melody_pcm_data.h`），零 RAM 占用

**播放流程**（在 `audio_th` 线程中执行）：
1. 收到 `BEEP_CMD_MELODY_START` 命令后，设置 `s_melody_running = RT_TRUE`
2. 打开 DAC，设置 8000Hz，音量取全局音量的 70%（更柔和）
3. 预填 2 份 PCM 数据到 DAC FIFO（消除段间间隙）
4. 循环：通过 `rt_sem_take(&s_audio_sem, timeout)` 以 200ms 为粒度等待约 4.1s，超时后补写 1 份 PCM
5. 收到 `BEEP_CMD_MELODY_STOP` 时，`s_melody_running` 置为 `RT_FALSE`，同时 `rt_sem_release` 立即唤醒线程中的超时等待
6. 关闭 DAC，线程回到等待命令状态

**关键设计**：
- `s_melody_running` 用 `volatile rt_bool_t` 保证跨线程可见
- 预填 2 份消除循环段间的静音间隙
- 停止时通过信号量立即唤醒（不再仅依赖轮询标志位延时），响应更快
- 音量取全局 70%，避免旋律盖过其它提示音

### 3.4 静音控制

语音交互期间（P37 长按录音→上传→等待识别结果），按键"滴滴"声会混入录音数据，干扰 ASR 识别。

**方案**：全局静音标志 `s_beep_muted`

| 时机 | 操作 |
|------|------|
| P37 长按、录音启动成功后 | `beep_set_muted(RT_TRUE)` — 抑制后续按键发声 |
| P37 松开、录音停止后 | `beep_set_muted(RT_FALSE)` — 恢复按键发声 |

`beep_play()` 和 `beep_melody_start()` 均在入口处检查静音标志，若静音则直接返回不发声。

### 3.5 音量控制

```c
void beep_set_volume(int volume);  // 范围 0~100，默认 65
```

- 当前实现为占位函数（仅钳位参数范围），后续可接入 `CODEC_CMD_SET_VOLUME` 设置 DAC 硬件音量
- 旋律音量自动取全局音量的 70%（在调用 DAC 写入前逻辑控制）

## 4. 与按键模块的集成

### 4.1 按键发声

在 `button_control.c` 的 `button_press_down()` 回调中：

```c
static void button_press_down(void *param)
{
    uint8_t gpio = button_get_trigger_gpio(param);

    // P13（休眠键）不发声
    if (gpio == BUTTON_GPIO_P13)
        return;

    // 等待识别结果或打印中不发声
    if (!audio_result_wait_is_locked() && !thermal_printer_is_printing())
        beep_play();
}
```

**锁判断逻辑**：与单击/长按回调保持一致——`audio_result_wait_is_locked()` 或 `thermal_printer_is_printing()` 任一为真时，不发声也不执行业务逻辑，避免打断关键流程。

### 4.2 静音与录音联动

在 `button_long_press_start()`（P37 长按）和 `button_press_up()`（P37 松开）中：

```c
// P37 长按 → 录音启动成功后
beep_set_muted(RT_TRUE);

// P37 松开 → 录音停止后
beep_set_muted(RT_FALSE);
```

### 4.3 系统初始化

`main.c` 在系统启动阶段调用：

```c
if (beep_init() != 0)
    rt_kprintf("beep_init failed\n");
```

`beep_init()` 创建信号量 `"beep_sem"` 并启动常驻 `"audio_th"` 线程，预分配蜂鸣 PCM 缓冲区。

## 6. 注意事项与已知约束

1. **DAC 独占**：蜂鸣和旋律共用同一个 `"sound"` DAC 设备，由统一音频线程的命令互斥机制保证同一时刻只有一个在播放。

2. **无 MP3 依赖**：本方案完全基于 PCM 直灌，不依赖 `minimp3` 或任何 MP3 解码器，**没有栈溢出风险**。此前 MP3 提示音方案已因 `minimp3` 栈上数十 KB 的 `mp3dec_scratch_t` 导致 `auprompt` 线程栈溢出死机而被移除。

3. **线程模型**：单一常驻 `"audio_th"` 线程（栈 4096B，优先级 27），低于音频上传等关键业务线程，不会抢占关键资源。

4. **Flash 占用**：旋律 PCM 数据 68.8KB（34400×2），若需裁剪可设置 `CFG_BEEP_MELODY_ENABLE=0`，或替换为更短的旋律。

5. **音量控制**：`beep_set_volume()` 当前为占位实现，未实际控制 DAC 硬件音量。后续可接入 `CODEC_CMD_SET_VOLUME` 生效。
