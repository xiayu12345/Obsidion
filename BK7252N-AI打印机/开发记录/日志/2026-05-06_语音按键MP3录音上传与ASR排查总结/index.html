    语音按键、MP3 提示音、录音上传与 ASR 排查总结¶

    本文汇总本期围绕 P37 长按语音链路、嵌入式 MP3 提示音、固件与 applications-对比 差异、以及 云平台语音识别异常 的讨论与结论，便于后续维护与排障。

    1. 需求与迭代过程¶

    阶段 内容
    早期 长按提示音曾尝试合成 PCM「叮咚」、长 MP3（flash 体积原因收紧）、再到 8 kHz 短 MP3（如基于「华夏不灭之光-02」转码）。
    故障 长按后出现 死机、听不到提示音；日志出现 auprompt 栈溢出、undefined instruction，继而 data abort、复位。
    表达误差 一度要求「去掉长按录音播放声音」：实际意图为 仅去掉 MP3 播放，保留 P37 长按录音上传与进二级桌面。
    纠正 恢复 audio_upload_start/stop、二级桌面、PRESS_UP；不再调用 audio_prompt_play_async()；移除 audio_prompt / minimp3 / prompt_mp3_data.* 等仅服务 MP3 的模块以减体积与风险。

    2. MP3 提示音死机：现象与根因分析¶

    2.1 现象¶

    • 线程 auprompt：stackoverflow=1，栈使用率约 71%（16 KB 栈配置下）。
    • undefined instruction（UND），PC 落在疑似被破坏的地址附近。
    • 随后 data abort、看门狗复位；提示音未播放完成。

    2.2 技术原因（结论）¶

    • minimp3 在 mp3dec_decode_frame() 内声明 mp3dec_scratch_t scratch（含 maindata、grbuf、syn 等），单次解码在栈上可达约十余 KB 量级。
    • 再加上 MINIMP3_MAX_SAMPLES_PER_FRAME 级 PCM 缓冲、调用链与 RT-Thread 上下文，16 KB 线程栈明显不足，易发生 栈溢出。
    • 溢出破坏 LR/PC 后，表现为 非法指令 / 异常向量连环触发，与日志一致。

    2.3 若将来恢复 MP3（可选方向）¶

    1. 显著增大 AUDIO_PROMPT_THREAD_STACK（例如 32 KB～48 KB），或
    2. 使用 MINIMP3_USE_STATIC_SCRATCH（在 minimp3.h 内对 scratch 使用 static，单线程解码场景下避免把大块结构放在栈上），并把 PCM 改为静态/堆分配；
    3. 或改为 离线转 PCM 嵌入，去掉解码器以根除栈压力。

    说明： 当前仓库已按产品方向 去掉 MP3 提示音模块，上述为备忘。

    3. 按键与录音链路（当前目标行为）¶

    • P37 长按：MQTT 在线检查 → audio_upload_set_mode → audio_upload_start → 进入二级桌面；不再播放 MP3。
    • P37 松开：audio_upload_stop。
    • P36 长按：二维码预览等（与对比目录老版相比为新增能力）。

    实现文件：applications/button_control/button_control.c（含 mqtt_backend_client.h 等）。

    4. 云平台 ASR「识别跑飞」：排查结论（结合 applications-对比）¶

    4.1 与 applications-对比 逐项对比结论¶

    在 applications/audio/ 与 applications/mqtt_backend/mqtt_backend_client.c 等与上传直接相关的部分：

    文件 结论
    audio_upload.h 与对比目录 一致（默认采样率仍为 8000 等）。
    audio_upload_payload.c 一致（Base64、JSON、mqtt_backend_publish(..., qos=0)）。
    audio_upload.c 编码与分包逻辑一致：20 ms 帧、opus_encode、每帧前 8 字节大端头（长度 + final_range）、约 50 帧一组 audioChunk、尾包 is_last。差异主要在 UI：lv_desktop_voice_set_state、发布失败提示、从未发出首包时的「过短」分支等。
    mqtt_backend_client 异步发布、单条 payload 4 KB 上限等与对比目录一致。

    因此：若「正常老固件」等同于 applications-对比 中的 applications，则 ASR 异常难以归因于「当前 applications 与对比目录在录音→Opus→MQTT 实现不一致」；更应怀疑 对比目录未包含的变更（驱动、LVGL 页面、lv_example_meter、板级配置、负载）或 云端解析假设与固件封装是否一致（见下）。

    4.2 仍需关注的协议与传输点（与新老目录无关的共性）¶

    1. 每帧 8 字节自定义头：若云端按 纯 Opus 裸流 拼接解码而未剥离该头，会导致解码失步 → 听感噪声、识别乱序/跑飞。需与后端确认是否约定剥离或使用统一封装格式。
    2. sample_rate 字段与真实解码：固件 JSON 中为 8000；若云端 忽略字段按 16 kHz 解码，会产生语速/音调异常（可与其它问题叠加）。
    3. MQTT QoS0 + 异步队列：存在 丢包、队列满、publish message too long 可能；应结合串口日志与云端 task_id / seq 是否连续排查。
    4. s_read_buf 仅 4 KB：最坏情况下多帧编码偏大存在 写穿理论风险，可通过日志或静态审查 All_Len 验证。

    5. 相关源码路径（备忘）¶

    用途 路径
    按键 / P37 录音 applications/button_control/button_control.c
    录音线程 / Opus applications/audio/audio_upload.c
    上传 JSON / Base64 applications/audio/audio_upload_payload.c
    采样率等宏 applications/audio/audio_upload.h
    MQTT 异步发布 applications/mqtt_backend/mqtt_backend_client.c
    老 applications 快照 applications-对比/（仅作 diff 基准，不一定等于线上「老镜像」）
    MP3（已移除） 曾包含 applications/audio/audio_prompt.c、minimp3.h、prompt_mp3_data.*；脚本可选保留 tools/scripts/embed_prompt_mp3.py

    6. 后续建议（短清单）¶

    1. ASR 仍异常：用 真实量产老 ELF/源码 与当前树做 全仓库 diff（不仅 applications）；抓取一段 opus_chunk_b64 解码对比头部模式。
    2. 与后端对齐：8 字节帧头是否需剥离；采样率以 JSON 为准还是写死 16 k。
    3. 传输：关键 audioChunk 是否改 QoS1 或确认队列深度；串口监控 publish queue full / too long。
    4. 若恢复 MP3：优先 静态 scratch + 足够栈 或 PCM 直灌，避免再次栈溢出。
