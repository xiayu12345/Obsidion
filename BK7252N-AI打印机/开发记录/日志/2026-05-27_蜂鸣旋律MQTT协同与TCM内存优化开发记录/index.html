    蜂鸣/旋律 MQTT 协同与 TCM 内存优化开发记录¶

    日期：2026-05-27
    工程：bk7252n_sdk-SDK_3.0.76.1_witch_lcd_bz
    编译命令：scons --beken=bk7252n -j8
    前置文档：[[项目/BK7252N-AI打印机/开发记录/日志/2026-05-18_按键发声蜂鸣与等待旋律开发记录]]（初版 beep 模块与旋律架构）

    1. 背景¶

    在初版「按键蜂鸣 + 生图等待旋律」落地后，联调 MQTT 语音生图链路时出现一系列问题：

    现象 影响
    云端 volumeSet 后按键「滴滴」音量不变 用户误以为音量设置无效
    旋律音量调节后出现沙沙声、卡顿 体验差，疑似音频链路异常
    MQTT 同步收 audioChunk 时旋律断续 生图等待 BGM 不稳定
    日志 rx dispatch failed 误导排障（实际收包成功）
    create print thread failed 收图完成后无法启动 mqtt_prn 打印线程，业务中断

    根因集中在：音量策略未分离、DAC DMA 异步复用同一块缓冲、大数组常驻 BSS 挤占 TCM 堆、MQTT 收包线程长时间占用 CPU。

    2. 改动文件清单¶

    文件 改动类型 说明
    applications/button_control/beep.c 修改 按键/旋律音量分离；DMA 乒乓 + tx_complete 背压；旋律缓冲改为运行时分配；缩小 chunk/缓冲深度
    applications/button_control/beep.h 修改 beep_set_volume() 文档：仅影响等待旋律
    applications/mqtt_backend/mqtt_backend_message.c 已有 volumeSet → beep_set_volume()
    applications/mqtt_backend/device_config.c 已有 配置加载时同步 beep_set_volume()
    applications/main.c 已有 启动时从 device_config_get_volume() 初始化旋律音量
    applications/audio/audio_result_wait.c 已有 等待生图时 beep_melody_start/stop
    applications/mqtt_backend/binary_rx/mqtt_backend_binary_audio_chunk.c 修改 收包/写 Flash 路径 rt_thread_yield()；收图完成前 beep_melody_stop() + 延时；修正返回值
    applications/mqtt_backend/mqtt_backend_async_rx.c 修改 同步收包模式 dispatch 前后 yield；audioChunk 成功路径不再误报失败
    applications/mqtt_backend/mqtt_backend_config.h 配置 CFG_MQTT_BACKEND_ASYNC_RX=0（大图不能异步队列，防 OOM）
    applications/mqtt_backend/mqtt_backend_client.c 修改 Paho 线程优先级下调，为 audio_th 让出 CPU
    applications/mqtt_backend/binary_rx/mqtt_backend_binary_rx.c 相关 mqtt_prn 线程创建（栈 6144）

    3. 问题与根因¶

    3.1 音量：volumeSet 对按键无效¶

    • beep_set_volume() 原先同时影响按键蜂鸣与旋律，或仅改软件变量未区分通路。
    • 产品要求：按键固定 65；旋律跟随配置文件 / volumeSet（0~100）。

    3.2 旋律失真、卡顿（DMA 指针复用）¶

    • rt_device_write() 在 DAC 驱动中只保存缓冲区指针，由 GDMA 异步读取（GDMA_X_SRC_DTCM_RD_REQ）。
    • 早期单块 s_stereo_chunk 在循环中被覆盖 → 杂音、断续。
    • 曾用 s_stereo_chunks[8][1024] 做乒乓，但 16KB 常驻 BSS。

    3.3 TCM 堆耗尽 → 打印线程创建失败¶

    BK7252N 内存模型（drivers/board.h + link.lds）：

    [ RAM: .data + .bss ... ] → _empty_ram → [ TCM 堆 → 256KB 末端 ]
    • rt_malloc / rt_thread_create 栈均走 TCM 堆（RT_KERNEL_MALLOC → tcm_malloc）。
    • s_stereo_chunks 占 0x4000（16KB）BSS，直接压缩 _empty_ram 之后的可用堆。
    • 收图峰值：MQTT 缓冲 + LVGL + mqtt_prn 栈（6144）+ 其它线程 → rt_thread_create("mqtt_prn", ...) 失败。

    out/rtthread.map 摘录（优化前）：

    .bss.s_stereo_chunks    0x00408488    0x4000    beep.o
_empty_ram              0x004292b8

    3.4 MQTT 同步收包与旋律争抢 CPU¶

    • CFG_MQTT_BACKEND_ASYNC_RX=0：Paho 回调内同步处理 audioChunk，单次处理可达数百毫秒。
    • 若 audio_th 优先级过低，无法及时 write / 响应 tx_complete，旋律 underflow。

    3.5 日志误报 rx dispatch failed¶

    • audioChunk 处理函数在「无需 JSON 应答」场景曾返回 -1。
    • mqtt_backend_async_rx_push() 在同步模式下将 dispatch != 0 视为失败并打印错误，实际收包与 Flash 写入已成功。

    4. 技术方案¶

    4.1 按键与旋律音量分离¶

    #define BEEP_FIXED_VOLUME   65   /* 按键蜂鸣，不随 volumeSet 变化 */

static int s_melody_volume = 65; /* 旋律，beep_set_volume / 配置文件 */

void beep_play(void) {
    beep_apply_fixed_codec_volume(dac);  /* CODEC_CMD_SET_VOLUME = 65 */
    ...
}

void beep_melody_start(void) {
    melody_apply_codec_volume(dac);      /* volume * 99 / 100，钳位 0~99 */
    ...
}
    • 不再在旋律循环内对每块 PCM 做软件缩放（曾引入沙沙声）。
    • 音量仅通过 Codec 硬件 CODEC_CMD_SET_VOLUME 设置。

    4.2 旋律 DMA 乒乓与背压¶

    参数 初版（问题版） 当前值
    CHUNK_MONO 512（32ms×2 立体声点） 256
    MELODY_BUF_COUNT 8 4
    缓冲存放 static int16_t s_stereo_chunks[8][1024] rt_malloc 池，停止后 rt_free
    常驻 RAM 16KB BSS 0（空闲时）
    播放中 RAM 16KB 4KB（4×256×2×sizeof(int16_t)）

    流程要点：

    1. melody_tx_done：tx_complete 时 rt_sem_release(&s_tx_done_sem)。
    2. melody_write_one_chunk：当 pipeline >= MELODY_BUF_COUNT - 1 时 rt_sem_take 等待 DMA 消费，避免覆盖在途缓冲。
    3. 启动时预填 MELODY_BUF_COUNT - 2 块，降低起播间隙。
    4. mono → stereo 展开：chunk[j*2] = chunk[j*2+1] = MELODY_PCM[...]。

    4.3 TCM 内存优化（本次核心）¶

    策略：旋律缓冲仅在播放期间从 TCM 堆分配，停止后释放，避免 16KB 永久占用 BSS。

    static int16_t *s_stereo_chunk_pool = RT_NULL;

static rt_bool_t melody_chunks_alloc(void) {
    s_stereo_chunk_pool = (int16_t *)rt_malloc(MELODY_POOL_BYTES);  /* 4096 */
    ...
}

static void melody_chunks_free(void) {
    rt_free(s_stereo_chunk_pool);
    s_stereo_chunk_pool = RT_NULL;
}
    • BEEP_CMD_MELODY_START 路径：alloc → 循环写 DAC → close → free。
    • 收图完成路径（mqtt_backend_binary_audio_chunk.c）在创建打印线程前调用 beep_melody_stop() + rt_thread_mdelay(100)，确保音频线程退出循环并 释放 4KB 池，再 rt_thread_create("mqtt_prn", 6144, ...)。

    效果：永久释放约 16KB TCM 堆；播放中峰值从 16KB 降至 4KB。

    4.4 线程优先级与 MQTT 协同¶

    线程 优先级（数值越小越高） 说明
    audio_th 14 须高于 Paho / 收包路径，及时喂 DMA
    Paho MQTT 20（下调） 避免长时间霸占 CPU
    mqproc 20 仅 CFG_MQTT_BACKEND_ASYNC_RX=1 时存在

    同步收包优化（mqtt_backend_async_rx.c）：

    • mqtt_backend_rx_dispatch_payload() 入口、出口各 rt_thread_yield()。
    • mqtt_backend_binary_audio_chunk.c 写 Flash 循环内 rt_thread_yield()。

    异步 RX 保持关闭：CFG_MQTT_BACKEND_ASYNC_RX=0。大图分片若入队会额外 malloc，易 OOM。

    4.5 收图 → 打印衔接¶

    /* mqtt_backend_binary_audio_chunk.c — result 帧处理末尾 */
beep_melody_stop();
rt_thread_mdelay(100);

if (mqtt_backend_binary_spawn_print_thread(...) != 0) {
    rt_kprintf("mqtt backend create print thread failed\r\n");
    ...
}

    4.6 日志与返回值¶

    • mqtt_backend_binary_audio_chunk 正常结束 return 0，避免同步 push 误报 rx dispatch failed。
    • audioChunk 无 MQTT JSON 应答属正常，不应返回失败码给 dispatch 层。

    5. 内存与资源一览¶

    5.1 旋律相关¶

    资源 位置 大小 说明
    MELODY_PCM[] Flash（melody_pcm_data.h） ~68KB 34400×int16，不占用 RAM
    beep_buf TCM 堆（audio_th 内 rt_malloc） ~2.4KB 按键 PCM，线程生命周期常驻
    s_stereo_chunk_pool TCM 堆（旋律播放时） 4KB 停止后释放
    audio_th 栈 TCM 堆 4096 创建时分配

    5.2 典型峰值（收图打印时）¶

    • 旋律已停、缓冲已释放 → TCM 可用于 mqtt_prn（栈 6144）及运行时 MQTT/LVGL 缓冲。
    • 若仍 OOM：用 list_memheap / map 继续排查其它大块分配（如 print_buf、LVGL draw_buf）。

    6. 业务集成关系¶

    语音上传完成 → audio_result_wait_start()
              → beep_melody_start()     // 循环 C 大调琶音

MQTT audioChunk 分片收包（同步，yield）
              → Flash 落盘

result 帧 OK  → beep_melody_stop()
              → mdelay(100)
              → mqtt_prn 打印线程

超时/失败     → audio_result_wait_clear() → beep_melody_stop()
    API / 配置 作用
    beep_play() 按键「叮咚」，Codec 音量 65
    beep_set_volume(v) 仅旋律；volumeSet / device_config / main 初始化
    beep_set_muted() P37 录音期间静音按键与旋律
    beep_melody_start/stop() 生图等待 BGM
    CFG_BEEP_MELODY_ENABLE 编译期裁剪旋律（默认 1）

    7. 注意事项与已知约束¶

    1. DMA 源地址须在 DTCM/TCM 可达范围：旋律池使用 rt_malloc（TCM 堆），满足 GDMA_X_SRC_DTCM_RD_REQ；不宜改为 SDRAM 除非驱动支持且实测通过。

    2. 缓冲变浅：CHUNK_MONO=256、MELODY_BUF_COUNT=4 后，MQTT 同步大包收图时旋律抗抖动余量小于旧版 8×512。若现场仍卡顿，可仅将 MELODY_BUF_COUNT 提到 6（播放时约 6KB），不要恢复 16KB 静态 BSS。

    3. DAC 独占：蜂鸣与旋律仍共用 "sound" 设备，由 audio_th 命令互斥。

    4. 打印线程：mqtt_prn 栈 6144；创建失败优先查 TCM 剩余空间，而非误认为 MQTT 协议错误。

    5. 重复源文件：曾存在 applications/mqtt_backend/mqtt_backend_binary_rx.c 与 binary_rx/ 下重复实现，已删除父目录副本，以 binary_rx/mqtt_backend_binary_rx.c 为准。

    8. 验证建议¶

    1. 音量：volumeSet 只改变等待旋律；按键声保持 65。
    2. 旋律质量：生图等待全程无规律杂音；停止后无残留 DAC 占用。
    3. 收图打印：日志出现 binary result ok 后应能 print thread dispatched，无 create print thread failed。
    4. 误报：不应再因 audioChunk 成功而出现 rx dispatch failed（除非真正 dispatch 返回非 0）。
    5. 内存：对比烧录前后 list_memheap 或 map 中 _empty_ram 偏移，确认 BSS 减少约 16KB。
